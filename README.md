단순 기능 검증을 넘어 유지보수를 고려한 최적의 스크립트 작성을 목표로 합니다.

# Playwright(Python) 자동화 테스트 프로젝트 소개
POM구조를 적용하여 E2E 자동화 테스트를 작성했습니다.

테스트케이스를 작성한 후 테스트 우선순위를 선정하여 설계하였습니다.

기능은 다양한 케이스로 확대해 나갈 예정입니다.

## 테스트 목적
정상적인 동작을 기준으로
잘못된 검증 결과 잡아내기 위해 예외사항 구문으로 설계하였습니다.
비정상 입력(XSS, SQL Injection),
로그인 이후 상태 및 접근 제어(로그아웃, 뒤로가기 등)를 검증했습니다

## 테스트 설계 근거
로그인 기능은 서비스의 핵심 진입점이므로, 단순 성공/실패뿐 아니라 다음을 중점적으로 검증했습니다

 - 인증 실패 시 시스템의 일관된 에러 처리
 - 로그인 이후 세션 상태 유지 여부
 - 비정상 입력에 대한 보안 취약성 여부
   
## 테스트케이스
  ##로그인 케이스 내용
- 정상 로그인 테스트를 수행했습니다
- 잘못된 비밀번호 입력을 테스트했습니다
- 존재하지 않는 계정으로 로그인 시도를 테스트했습니다
- username 미입력 상태를 테스트했습니다
- password 미입력 상태를 테스트했습니다
- username/password 모두 미입력 상태를 테스트했습니다
- 비정상 형식의 username 입력을 테스트했습니다
- locked 계정 로그인 시도를 테스트했습니다
  
  ##보안테스트
- SQL Injection 입력('OR '1'='1)을 테스트했습니다
- XSS 입력(<script>alert(1)</script>)을 테스트했습니다
  
  ##예외 케이스
- 로그아웃 이후 inventory 페이지 접근 차단 여부를 테스트했습니다
- 로그아웃 이후 뒤로가기 시 접근 가능 여부를 테스트했습니다
  
  ##성능 테스트
- 로그인 응답 시간을 측정했습니다 (5초 기준)
- 
## 테스트 기준
- 로그인 성공: inventory 리스트가 표시되는지 확인했습니다
- 로그인 실패: 에러 메시지 텍스트를 검증했습니다
- 응답 시간: 5초 이하 기준으로 설정했습니다
- 
## 회고
  단순히 자동화를 구현하는 것에서 끝나는 것이 아니라, 어떤 상황을 테스트해야 하는지를 고민하는 과정이 중요하다고 느꼈습니다.
  특히 로그인 이후의 상태나 예외 케이스를 고려하는 것이 테스트 품질에 큰 영향을 준다고 생각했습니다.
